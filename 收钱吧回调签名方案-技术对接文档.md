## 三、签名验证方案

Claw 平台对接两种回调类型，签名方式不同：

### 3.1 WAP 扫码支付回调（推荐，微信/支付宝扫码）

**签名算法：MD5**
**签名位置：URL 参数 `sign`**

#### 参与签名的参数（按字典序排列）

```
client_sn=CLAW1234567890123&
notify_url=https://claw-backend-2026.onrender.com/api/webhook/shouqianba&
operator=claw_admin&
return_url=https://claw-frontend.pages.dev/payment/result&
subject=Claw 基础版 - 30天&
terminal_sn=100111220054389553&
total_amount=19900&
key=<终端密钥 terminalKey>
```

#### 签名计算公式

```
sign = MD5( 参数字符串 &key=<terminalKey> ).toUpperCase()
```

#### 示例（伪数据）

```
// 1. 所有参数按 key 字典序排序
// 2. 拼接成 query string 格式（无空格）
// 3. 末尾追加 &key=终端密钥
// 4. MD5 → 大写

sign_str = "client_sn=CLAW1234567890123&notify_url=https://claw-backend-2026.onrender.com/api/webhook/shouqianba&operator=claw_admin&return_url=https://claw-frontend.pages.dev/payment/result&subject=Claw 基础版 - 30天&terminal_sn=100111220054389553&total_amount=19900&key=677da351628d3fe7664321669c3439b2"
sign = MD5(sign_str).toUpperCase()
// sign = "A1B2C3D4E5F6..." (示例)
```

#### 验签实现代码（Node.js）

```javascript
const crypto = require('crypto');

function verifyNotifySign(params, terminalKey) {
  // 排除 sign 和 sign_type 本身
  const { sign, sign_type, ...signParams } = params;

  // 按 key 字典序排序
  const sortedKeys = Object.keys(signParams).sort();

  // 拼接参数字符串
  const pairs = sortedKeys.map(k => `${k}=${signParams[k]}`);
  const signStr = pairs.join('&') + '&key=' + terminalKey;

  // 计算 MD5 并大写
  const expectedSign = crypto
    .createHash('md5')
    .update(signStr)
    .digest('hex')
    .toUpperCase();

  return expectedSign === (sign || '').toUpperCase();
}
```

### 3.2 Card 支付回调（RSA）

**签名算法：RSA-SHA256**
**签名位置：HTTP Header `Authorization`**

Card 类支付（银行卡/信用卡）回调通过 RSA 公钥验签：

```javascript
const crypto = require('crypto');

function verifyRsaSign(bodyStr, authHeader) {
  if (!authHeader || !authHeader.startsWith('sn ')) {
    return false;
  }
  const rsaSign = authHeader.substring(3); // 去掉 "sn " 前缀
  const expected = crypto
    .createHash('sha256')
    .update(bodyStr)
    .digest('base64');
  return rsaSign === expected;
}
```

## 四、三层状态判断机制

> ⚠️ **重要**：Claw 平台采用三层状态判断，需收钱吧确保回调按以下顺序推送状态。

### 4.1 第一层：通讯层（HTTP 状态）

检查 `result_code` 字段：

| 值 | 含义 | 处理方式 |
|----|------|---------|
| `SUCCESS` | 通讯正常，继续 | 进入第二层 |
| `FAIL` / 其他 | 通讯失败 | **不更新订单**，返回 `success`（避免重推） |

### 4.2 第二层：业务结果层

检查 `biz_response.result_code` 字段：

| 值 | 含义 | 处理方式 |
|----|------|---------|
| `SUCCESS` | 业务处理成功 | 进入第三层 |
| `ERROR` / 其他 | 业务失败 | 记录错误信息，返回 `success` |

回调示例（业务失败）：
```json
{
  "result_code": "SUCCESS",
  "biz_response": {
    "result_code": "ERROR",
    "error_message": "余额不足"
  }
}
```

### 4.3 第三层：订单状态层

检查 `order_status` 字段：

| 值 | 含义 | 处理方式 |
|----|------|---------|
| `PAID` | 支付成功 | 更新订单为 `paid`，激活会员 |
| `CLOSED` | 订单超时关闭 | 更新订单为 `closed` |
| `REFUND` | 已退款 | 更新订单为 `refunded` |

回调示例（支付成功）：
```json
{
  "result_code": "SUCCESS",
  "biz_response": {
    "result_code": "SUCCESS"
  },
  "client_sn": "CLAW1234567890123",
  "order_status": "PAID",
  "total_amount": 19900,
  "payway": "wechat",
  "paid_time": "2026-04-23T20:00:00+08:00"
}
```

## 五、回调响应规范

| 场景 | HTTP 状态码 | 响应 Body | 说明 |
|------|------------|----------|------|
| 验签成功 + 订单已存在 | 200 | `"success"` | 处理完成 |
| 验签失败 | 200 | `"fail"` | 收钱吧将按策略重试 |
| 订单不存在 | 200 | `"success"` | 幂等处理，不重试 |
| 服务器异常 | 200 | `"success"` | **必须返回 success**，避免收钱吧持续重推 |

> ⚠️ **重要**：无论订单是否存在或处理结果如何，服务器**必须返回 HTTP 200 + "success"**，否则收钱吧将持续重推回调，导致重复处理。

## 十、常见问题

### Q1: 回调没有收到？
检查收钱吧商户后台的回调地址是否配置正确，确认公网可访问（Render 后端需通过 Health Check）。

### Q2: 验签失败？
确认 `terminalKey` 与收钱吧后台一致，注意签名字符串的排序方式（字典序）和结尾 `&key=` 格式。

### Q3: 订单已支付但会员未激活？
检查三层状态判断是否完整，特别是 `biz_response.result_code` 是否为 `SUCCESS`。

### Q4: 如何处理重复回调？
Claw 平台在更新订单前检查 `status === 'paid'`，已支付订单直接返回 `success` 不再处理。
